# 方向一：运维/SRE Agent — 执行方案

## 产品定位

**"凌晨三点的搭档"** — 一个懂你系统历史的本地 AI，专门处理线上问题。

不是通用编程助手，而是深度绑定你的基础设施、历史事故、运维知识的专属 agent。

---

## 核心价值主张

- 记住每一次故障的原因和解法，下次同类问题秒级响应
- 自动关联告警 → 日志 → 历史事故 → runbook，给出处置建议
- 本地运行，生产环境日志/配置不出本机

---

## 目标用户

- 独立开发者（自己维护线上服务）
- 小团队 SRE / DevOps（2-10 人）
- 有 on-call 压力的后端工程师

---

## 核心功能模块

### 1. 事故记忆系统
- 每次故障处理后自动提取：症状、根因、解法、影响范围
- 向量化存储，下次相似告警时自动召回相关历史
- 支持手动标注和补充

### 2. 知识库
- 导入 runbook（Markdown/Confluence/Notion 导出）
- 导入架构图描述（文本形式）
- 自动从对话中提取操作步骤沉淀为 runbook
- 监听本地 runbook 目录自动索引

### 3. 日志分析 Agent
- 接受日志片段输入，自动识别异常模式
- 关联知识库中的已知问题
- 给出排查步骤建议（不自动执行，需确认）

### 4. 告警响应 Agent
- 接受告警 webhook 或手动粘贴告警内容
- 自动检索：历史相似事故 + 相关 runbook + 近期变更
- 输出结构化的处置建议：可能原因（按概率排序）+ 建议操作步骤

### 5. 命令执行（受控）
- 预定义安全命令白名单（kubectl get、docker ps、tail log 等）
- 危险命令（delete、restart、scale）需要二次确认
- 所有执行记录存档

### 6. 事后复盘辅助
- 根据本次事故的操作记录自动生成 Post-Mortem 草稿
- 提取 Action Items 并追踪

---

## 技术架构

```
输入层:
  - CLI（主入口）
  - Webhook 接收器（告警推送）
  - 文件监听（日志文件 tail）

Agent 层:
  - AlertAgent：告警分析 + 历史关联
  - LogAgent：日志模式识别
  - RunbookAgent：步骤检索 + 执行建议
  - PostMortemAgent：复盘文档生成

记忆层:
  - 事故记忆（SQLite + 向量）
  - 操作历史（SQLite）

知识库:
  - Runbook 索引（向量检索）
  - 架构知识（结构化 + 向量）

执行层:
  - 命令白名单执行器
  - 确认机制（危险操作拦截）
```

---

## MVP 功能范围（6-8 周）

**Week 1-2：基础框架**
- CLI 入口
- LLM 接入（OpenAI 兼容）
- SQLite 存储层

**Week 3-4：核心记忆**
- 事故记录 CRUD
- 向量化 + 相似度检索
- 手动记录事故流程

**Week 5-6：日志/告警分析**
- 日志片段分析 Agent
- 历史事故关联
- 处置建议生成

**Week 7-8：知识库 + 执行**
- Runbook 导入 + 检索
- 受控命令执行
- Post-Mortem 生成

---

## 差异化壁垒

1. **记忆积累效应**：用得越久，历史事故越多，响应越准确，竞品无法复制
2. **本地隐私**：生产日志、配置、密钥不出本机，企业用户核心诉求
3. **垂直场景**：OpenCode/Cursor 不做运维，PagerDuty/OpsGenie 不做 AI 分析

---

## 商业化路径

- 个人版：免费开源（建立用户基础）
- 团队版：共享事故知识库 + 多人协作（$20/人/月）
- 企业版：私有化部署 + 权限管理 + 审计日志

---

## 风险点

- 用户对 agent 执行生产命令的信任建立需要时间
- 冷启动问题：没有历史事故时价值有限，需要引导用户快速录入
- 日志格式多样，解析层需要较强的适配能力
