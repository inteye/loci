# 方向三：代码库理解 Agent

## 产品定位

`loci` 的核心不是“帮你写代码”，而是“帮你接手和读懂代码库”。

第一阶段只聚焦四件事：

1. `Explore`：这个模块/文件/符号是做什么的
2. `Trace`：为什么这么写，什么时候变成这样
3. `Navigate`：相关代码在哪里，谁依赖谁，谁调用谁
4. `Document`：把理解结果沉淀成可复用文档

通用任务执行、开放式技能平台可以保留，但不应主导产品定义。

---

## 当前设计原则

### 1. 以“项目认知层”为中心

系统需要一个统一的项目认知层，而不是图谱、记忆、知识库三套并列事实源。

- `Graph`：主存，保存项目事实
- `Memory`：只保存会话上下文和用户偏好
- `Knowledge`：外部材料和提炼结论，最终应回写到图谱

如果问答后的结论不能进入统一认知层，“越用越懂项目”的飞轮就不会成立。

### 2. 代码事实和推断分层

必须区分：

- 代码事实：文件、符号、调用关系、提交记录、时间线
- 模型推断：模块职责、设计意图、风险判断、文档摘要

推断必须带证据来源，避免把 LLM 输出直接当成事实。

### 3. 专用 Agent 优先

这个产品更适合专用 agent，而不是通用 planner。

- `ExploreAgent`：解释代码对象
- `TraceAgent`：构建设计决策时间线
- `NavigateAgent`：做定位、检索、依赖追踪
- `DocAgent`：生成模块文档、架构摘要、交接材料

CLI 和 API 应直接路由到这些能力，而不是先走一个泛化的 DAG 任务分解器。

---

## 核心数据模型

建议图谱至少覆盖以下节点和关系：

- 节点：`file`、`symbol`、`module`、`concept`、`commit`、`decision`
- 边：`contains`、`calls`、`implements`、`depends_on`、`changed_in`、`explained_by`、`related_to`

其中：

- `commit` 节点承载代码演进事实
- `decision` 节点承载“为什么这么做”的归纳结果
- `concept` 节点承载跨文件、跨模块的抽象知识

这样问答沉淀、git 溯源、文档生成才能落到同一套结构上。

---

## 能力边界

### 已适合承诺的能力

- 项目扫描和基础索引
- Rust 优先的符号提取和代码解释
- 基于图谱/向量/记忆的问答
- 基础 Git 历史读取

### 暂不应过度承诺的能力

- 深度多语言理解
- 完整设计决策还原
- 探索结果自动增长为高质量知识图谱
- 大规模代码库下已验证的性能表现

对外表达应把“已支持扫描”与“已支持深度理解”分开。

---

## 近期重构重点

### P0：把主链路收紧

1. 明确 `Graph` 是唯一项目事实主存
2. 问答后的知识提炼回写图谱，而不是只写 memory/knowledge
3. CLI 的 `ask / explain / diff / doc` 绑定到专用 agent 能力

### P1：补齐 Trace 闭环

1. 增加 `git blame`
2. 建立 `commit -> file/symbol` 关联
3. 输出“时间线 + 证据 + 结论 + 置信度”
4. 支持“这个函数为什么这么写”类问题的稳定回答

### P2：统一文档生成

1. 基于图谱生成模块说明
2. 生成 onboarding 文档
3. 生成交接文档
4. 明确文档中的事实和推断来源

---

## 风险与约束

- 多语言支持不能只靠统一 parser 抽象，最终仍要按语言补深度能力
- Git 溯源如果没有 blame、时间线和证据归因，就很难形成真正壁垒
- 如果继续同时推进通用 agent 与垂直产品，系统边界会持续漂移
- 在缺少评测集前，不应把“理解正确率”写成既成事实
