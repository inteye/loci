# 方向三：代码库理解 Agent — 执行方案

## 产品定位

**"接手任何代码库的第一天就像工作了一年"** — 专门解决"读懂陌生代码库"这个高频痛点。

不是代码生成工具，而是代码库的理解和导航工具。帮你快速建立对一个项目的全局认知，并把这个认知持久化。

---

## 核心价值主张

- 接手老项目不再需要两周才能搞清楚架构
- "这段代码为什么这么写"有答案（从 git history + 注释 + 上下文推断）
- 知识图谱随着你的探索自动积累，离职交接有完整文档

---

## 目标用户

- 接手遗留项目的开发者
- 参与开源项目的贡献者
- 需要快速理解第三方库源码的工程师
- 技术 Lead（需要给新人讲解系统）

---

## 核心功能模块

### 1. 代码库自动分析
- 扫描项目结构，识别：入口点、核心模块、数据流、依赖关系
- 生成架构摘要（文字描述 + 模块关系图）
- 识别关键路径（最重要的 10 个文件/函数）
- 支持语言：Rust、Python、TypeScript、Go、Java

### 2. 探索记忆
- 记录每次"问答"：你问了什么、agent 回答了什么、涉及哪些文件
- 自动构建项目知识图谱（实体：模块、函数、概念；关系：调用、依赖、实现）
- 知识图谱随探索自动增长，不需要手动整理

### 3. 代码溯源
- "这个函数为什么这么写" → 分析 git log + blame + 相关 issue/PR 引用
- "这个设计决策是什么时候做的" → 时间线还原
- 需要配合 git 仓库，离线也能用

### 4. 智能导航
- "找到处理用户认证的代码" → 语义搜索，不是关键词搜索
- "这个模块的所有调用方" → 依赖分析
- "哪里处理了这个错误码" → 跨文件语义理解

### 5. 文档生成
- 根据探索记录自动生成：模块说明、架构文档、新人入门指南
- 生成交接文档（"这个项目最重要的 10 件事"）
- 输出格式：Markdown，可直接放进 README 或 Wiki

### 6. 对比分析
- "这个项目和 X 项目的架构有什么区别" → 跨项目知识对比
- 适合同时维护多个项目的工程师

## 技术架构

```
输入层:
  - CLI（主入口）
  - 本地目录扫描
  - Git 仓库读取

分析层:
  - 静态分析器（AST 解析，各语言适配）
  - Git 历史分析器
  - 语义理解（LLM）

知识图谱:
  - 节点：文件、函数、模块、概念
  - 边：调用关系、依赖关系、演进关系
  - 存储：SQLite（结构化）+ 向量库（语义）

Agent 层:
  - ExploreAgent：回答"这是什么"类问题
  - TraceAgent：回答"为什么这样"类问题
  - NavigateAgent：回答"在哪里"类问题
  - DocAgent：生成文档

输出层:
  - 交互式问答（CLI）
  - Markdown 文档导出
```

## MVP 功能范围（6-8 周）

**Week 1-2：代码扫描基础**
- 项目结构分析（文件树 + 语言识别）
- 基础 AST 解析（函数/类提取）
- 向量化索引

**Week 3-4：语义问答**
- "这个文件/函数是做什么的" → LLM 分析
- 探索记录存储
- 基础知识图谱构建

**Week 5-6：Git 溯源**
- git log/blame 读取
- 变更历史关联
- "为什么这么写"推断

**Week 7-8：文档生成**
- 架构摘要自动生成
- 新人入门文档
- 交接文档模板

## 差异化壁垒

1. **知识积累效应**：探索越多，知识图谱越完整，理解越深，竞品无法复制
2. **Git 历史理解**：不只看当前代码，理解代码的演进过程，这是 Cursor/Copilot 没有的
3. **本地隐私**：代码不上传，企业私有代码库可用
4. **跨会话记忆**：上次探索的发现这次直接用，不用重新分析

---

## 与 OpenCode/Cursor 的区别

| | 本产品 | OpenCode/Cursor |
|---|---|---|
| 核心目标 | 理解代码库 | 生成/修改代码 |
| 记忆 | 持久知识图谱 | 无 |
| Git 历史 | 深度分析 | 基本感知 |
| 输出 | 理解 + 文档 | 代码 |
| 适用场景 | 读懂陌生代码 | 写新代码 |

---

## 风险点

- 静态分析质量依赖语言适配，多语言支持工作量大
- 大型代码库（百万行）的分析性能是挑战
- 知识图谱的准确性依赖 LLM 理解质量，可能有幻觉
- 冷启动：第一次分析需要时间，用户耐心有限
